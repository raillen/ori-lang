# Topic 2 - Advanced Generics And Generic Stdlib Shapes

## Current Gap

This topic covers the remaining gap between the current generic subset and the
desired generic runtime/stdlib surface.

Known areas to discuss:

- generic collection runtime support;
- generic element clone/destroy behavior;
- generic equality, hash, and ordering behavior;
- cross-type list HOFs;
- generic ordered collections;
- generic `lazy<T>`;
- diagnostics for unsupported type shapes;
- monomorphization scale limits.

## Maintainer Decision

- [x] Direction discussed in chat.
- [x] Final decision recorded here.
- [x] Follow-up docs identified.

Decision summary:

```text
Approved direction: capability-based generic model.

Generics in 0.4.2-beta.rc1 should expand through explicit compiler/runtime
capabilities instead of keeping only narrow specializations or attempting
universal generic support in one step.

Initial capability areas to design:
- clone/copy;
- destroy/release;
- equality;
- hash;
- ordering;
- transfer;
- optional debug/display if needed by a stdlib API.

Capability visibility decision:
- capabilities are internal compiler/runtime gates in `0.4.2-beta.rc1`;
- do not expose public traits such as `Hashable`, `Equatable`, `Cloneable`,
  or `Orderable` in this train;
- diagnostics may name the missing capability in user-facing language, but
  without committing to a public trait API;
- promotion to public traits or `@derive(...)` is a future design decision
  after the internal mechanism is validated.

Scope decision:
- use the medium scope for `0.4.2-beta.rc1`;
- implement the internal foundation for clone/copy, destroy/release, equality,
  hash, ordering, and transfer;
- include cross-type `map<T,U>` and generic `reduce<T,U>`;
- include `circbuf<T>`, `grid2d<T>`, and `grid3d<T>` for supported payloads;
- include `pqueue<T>`, `btreemap<K,V>`, and `btreeset<T>` for supported
  ordered/key shapes;
- keep `lazy<T>` and lazy iterators conditional: implement them only if the
  generic foundation is stable without forcing risky runtime redesign.

Implementation architecture decision:
- use a central semantic capability model;
- the checker calculates type capabilities once and records the result in the
  semantic model/HIR-facing type metadata;
- emitter/runtime code generation must consume this approved capability result
  instead of rediscovering capabilities independently;
- avoid checker-only validation with duplicated backend inference;
- avoid dynamic runtime vtables for every generic shape in this train unless a
  specific stdlib feature proves it is necessary.

Backend materialization decision:
- use a hybrid model;
- emit concrete helper functions per canonical type where needed;
- group helper pointers in internal ops tables for containers/runtime paths that
  need generic operations;
- ops tables are backend/runtime implementation details, not public API;
- deduplicate helpers and ops tables by canonical type key;
- never let the emitter approve a shape that the semantic capability model
  rejected.

Supported beta capability shapes:
- clone/destroy: primitive scalars, `text`, `bytes`, safe tuples, safe structs,
  and approved generic collection instantiations;
- equality: primitive scalars except non-finite-sensitive float use in key
  contexts, `text`, `bytes`, safe tuples, safe structs, and approved enum
  shapes without unsafe payload requirements;
- hash: `bool`, integer types, `text`, `bytes`, safe tuples/structs whose fields
  are hash-capable; float keys are rejected in this train;
- ordering: `bool`, integer types, and locale-neutral `text`; float ordering in
  ordered containers is rejected in this train because NaN/non-finite behavior
  should not leak into collection invariants;
- transfer: primitive scalars, `text`, `bytes`, and safe tuples/structs whose
  fields are transfer-capable.

Collection contract decisions:
- `map<K,V>` and `set<T>` require internal hash plus equality capability for
  keys/elements;
- `pqueue<T>`, `btreemap<K,V>`, and `btreeset<T>` require internal ordering
  capability;
- explicit user comparators are deferred;
- public `Hashable`/`Comparable` traits are deferred;
- compound public bounds are deferred; internal capability conjunctions are
  allowed.

Diagnostics decisions:
- unsupported shapes should fail during check, not at C compile time;
- diagnostics should say which operation needs which capability;
- diagnostic wording may use phrases such as "stable hash capability" or
  "stable ordering capability" without naming a public trait.

`lazy<T>` decision:
- `lazy<T>` is not a required exit criterion for Topic 2;
- implement it in this train only after the approved generic containers are
  green and without changing the runtime ownership model;
- lazy iterators remain deferred unless `lazy<T>` lands with low risk.

Follow-up docs:
- docs/spec/language/runtime-model.md
- docs/spec/language/stdlib-reference-by-topic.md
- docs/spec/language/post-v1-remaining-language-work.md
- a new or updated internal decision doc for generic runtime capabilities
```

Decision summary:

```text
Approved direction: pragmatic ABI expansion by approved shapes.

The goal is real C interop without turning FFI into a broad unsafe escape hatch.

Necessary for 0.4.2-beta.rc1:
- C representation marker for FFI-safe structs;
- FFI-safe struct-by-value arguments and returns;

- raw C varargs;
- direct managed values crossing C boundaries as loose pointers;
- mutable extern globals without an unsafe gate.

  to be documented as `attr repr("c")` unless parser constraints require a

- user structs cross by value only when explicitly marked C-repr and all fields
  are FFI-safe;
- `extern const` supports FFI-safe scalar/C-repr values only;
- mutable extern globals are not normal safe FFI; if implemented, they must be
  reachable only through an unsafe gate or `std.unsafe`-style API;
- conditional externs should be declarative and inspectable, not arbitrary build
  scripts;
- captured closure callbacks require C callback shape with user_data and an
  explicit cleanup hook;
- direct managed pointers are rejected; managed interop must use opaque handles
  or runtime-approved wrappers;
- raw varargs stay rejected; public APIs should use typed wrappers.
```

- package optional dependencies and feature flags.

## Maintainer Decision

- [x] Direction discussed in chat.
- [x] Final decision recorded here.
- [x] Follow-up docs identified.

Decision summary:

```text
Approved direction: foundation stdlib plus official packages.

The stdlib in 0.4.2-beta.rc1 should provide portable foundations and protocol
building blocks, not full application frameworks.

Approved boundaries:
- include `Stream<T>` / `Sink<T>` as foundational abstractions;
- expand `std.net` with listener/server and TCP stream adapters;
- add TLS as a provider/interface model with capability diagnostics;
- expand `std.http` as foundation APIs: request, response, method, status,
  headers, body, simple client/server behavior;
- treat WebSocket as a protocol foundation, not a framework;
- expand `std.time` calendar/date only if it stays small and portable;
- include generic `std.lazy<T>` only if Topic 2's generic foundation supports
  it safely;
- implement package feature flags and optional dependencies only where needed
  for platform/provider selection;
- keep middleware stacks, auth, sessions, templating, ORM, DB abstraction,
  image manipulation, and full web frameworks outside stdlib and in packages.

Detailed stdlib decisions:
- `Stream<T>` / `Sink<T>` should be push/adaptable foundation abstractions, not
  hidden async syntax;
- async delivery uses jobs/channels from Topic 3 as adapters;
- `std.net` should build on the existing `Connection` surface and add
  `Listener`, `listen`, `accept`, listener `close`, and TCP stream/sink
  adapters;
- `accept` should be timeout-aware or non-blocking by contract; hidden
  indefinite blocking is not the default teaching path;
- TLS ships first as an interface/provider boundary with capability diagnostics;
  a default provider is optional for this train;
- `std.http` stays a foundation module: request, response, method, status,
  headers, body, simple client, and simple server hooks;
- no stdlib middleware stack, auth/session model, template layer, ORM, or web
  framework in this train;
- WebSocket starts as protocol foundation and helpers after HTTP/TLS boundaries
  are stable; production framework behavior belongs in packages;
- `std.time` may add small portable calendar/date helpers, but no timezone
  database or locale-heavy calendar system in this train;
- package feature flags and optional dependencies should be implemented only to
  support explicit provider/platform selection, especially TLS and native
  integrations;
- package defaults must stay minimal and visible.

Follow-up docs:
- docs/spec/language/stdlib-model.md
- docs/spec/language/stdlib-reference-by-topic.md
- docs/spec/language/project-model.md
- docs/spec/language/post-v1-remaining-language-work.md
- docs/reference/stdlib/
```
