# Zenith 1.0.0-rc.1 Release Notes

Status: historical RC draft, not the current installable package line.

Current installable package line: `0.4.1-alpha.1` (2026-05-09).

This draft records the release-candidate direction discussed after the local
RC-public readiness work completed on 2026-05-08.

Planned public release:

- tag: `v1.0.0-rc.1`;
- commit: `3db2a30`;
- URL: `https://github.com/raillen/zenithlang/releases/tag/v1.0.0-rc.1`.

## What This RC Is For

Use this RC to validate the current public Zenith contract:

- reading-first syntax and diagnostics;
- explicit `optional<T>` and `result<T,E>` error flow;
- C backend as the current executable behavior oracle;
- core stdlib and documented stdlib subsets;
- project, package and CLI workflow;
- release-readiness evidence for sanitizer, Valgrind, performance and docs.

## Highlights

- No known local P0/P1 implementation blocker remains in the RC readiness
  reports.
- `std.collections` now has an explicit public contract:
  - `list<T>`, `map<K,V>` and `set<T>` are the main generic collection surface;
  - advanced structures are specialized in this RC.
- Public/reference docs were refreshed to avoid promising behavior beyond the
  current executable subset.
- Historical audit reports were marked as historical/superseded where they
  could be confused with current blockers.
- Sanitizer, Valgrind and full performance evidence are archived under
  `docs/internal/reports/audit/evidence/`.
- Installer/package orchestration exists for Windows and Linux, including
  `zt`, `zpm`, `zt-lsp`, `stdlib`, `runtime`, licenses, README and changelog.

## Accepted RC Limits

These are known limits, not hidden blockers:

- `grid2d<T>`, `grid3d<T>`, `pqueue<T>`, `circbuf<T>`,
  `btreemap<K,V>` and `btreeset<T>` are post-RC generic expansion work.
- `std.jobs`, `std.channels`, `std.shared` and `std.atomic` expose typed
  facades, but the executable runtime subset is still narrow.
- FFI is useful for the current extern-C subset, but it is not a general FFI
  system yet.
- Backend conformance beyond the C backend is future work.
- Registry, production LSP, published installers and public playground maturity
  are ecosystem work, not the core RC contract.
