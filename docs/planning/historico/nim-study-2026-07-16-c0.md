## 1. TL;DR (3 linhas)

O Nim faz RC **no header, inline e não-atômico**; a Ori faz RC **atômico via
lock global + HashMap** — o modelo Ori é mais seguro por default, mas cada
retain/release paga um lock. O caminho de destroy é estruturalmente igual
(dec → dtor por tipo → free → cascata), mas a Ori tem **duas** cascatas
(dtor de campos *e* edges) e precisa de auditoria para garantir que nenhum
tipo usa as duas ao mesmo tempo. O collector Nim é incremental sobre
**suspeitos** com threshold adaptativo; o da Ori é **full-heap scan** com
threshold fixo de 256 allocations.

## 2. O que o Nim faz (com trechos/caminhos)

### Header e RC (`lib/system/arc.nim`)

- `RefHeader` = **1 word** `rc: int`; sob ORC ganha `rootIdx: int` (remoção
  O(1) da lista de suspeitos). Header fica **antes** do payload
  (`head(p) = p - sizeof(RefHeader)`).
- O RC é guardado **deslocado** (`rcShift = 4` sob ORC): os 4 bits baixos do
  mesmo word guardam **cor** (black/gray/white, 2 bits), o bit sticky
  `maybeCycle` e `jumpStackFlag`. `rcIncrement = 0b10000`.
- `nimIncRef`/`nimDecRefIsLast` são inline, **não-atômicos** por default
  (atômico só em `--mm:atomicArc`/YRC). Justificativa: subgrafos são
  *movidos* entre threads, não compartilhados (`doc/mm.md`).
- **Não há ponteiro de destrutor no header**: o dtor vem do RTTI v2
  (`PNimTypeV2` no offset 0 do objeto; `rti.destructor`).

### Destroy (`arc.nim` + hooks de `doc/destructors.md`)

- O compilador emite `if nimDecRefIsLast(p): nimDestroyAndDispose(p)` — a
  decisão de destruir fica no **código gerado**, não no runtime.
- `nimDestroyAndDispose`: chama `rti.destructor` (o `=destroy` liftado, que
  desce campos e faz dec/destroy de cada ref filho) e só então
  `nimRawDispose` (free do bloco). **Campos antes do free do header.**
- 6 hooks por tipo: `=destroy`, `=wasMoved`, `=sink`, `=copy`, `=dup`,
  `=trace`. Move semantics: `lastReadOf(x)` via DFA transforma copy em move.

### Cycle collector ORC (`lib/system/orc.nim`)

- Papers no cabeçalho: **Bacon 2001** (trial deletion síncrono) + **Lins
  2008** ("critical links" / jump stack — desligado: `useJumpStack = false`).
- Candidato a ciclo: quando um decref **não** zera e o tipo não é `acyclic`,
  `rememberCycle` registra a cell em `roots` (threadvar). `rootIdx > 0`
  evita duplicata; incref/destroy tiram da lista.
- `registerCycle` dispara `collectCycles()` quando
  `roots.len - 128 >= rootsThreshold`. O threshold é **adaptativo**: se o
  passe libertou ≥ 50% do que tocou, threshold encolhe (÷3×2, min 16);
  senão cresce (×1.5). Atalho: se `rcSum == edges` durante `markGray`, tudo
  é lixo (coleta os gray direto, sem `scan`).
- Fases: `markGray` (decrementa trial via `=trace`, com trace stack
  explícita — sem recursão), `scan` (rc ≥ 0 → `scanBlack` restaura; senão
  white), `collectColor` (coleta whites, `entry[] = nil` antes do dtor para
  o dtor não tocar objeto moribundo).
- Proteção contra reentrância (bug #22927): durante os frees, `roots.len = 0`
  e threshold vai a `high(int)` — dtors podem registrar novos suspeitos sem
  corromper o passe.

## 3. O que a Ori faz hoje (caminhos)

Fonte: `compiler/crates/ori-runtime/src/lib.rs` (workspace atual).

- Header: `OriHeapHeader { refcount: AtomicI64, destructor: Option<fn> }`
  (16 bytes) — o **dtor mora no header**, registrado por alocação em
  `ori_alloc` (`lib.rs:24`, `lib.rs:266`).
- Estado global `ARC_STATE` (Mutex): `allocations: HashMap payload→(header,
  size)` + `edges` duplamente indexadas (`by_owner`/`by_child`).
- `ori_arc_retain`/`release` (`lib.rs:294`, `lib.rs:312`): **lock global +
  lookup no HashMap** para validar o ponteiro (ignora não-managed em
  silêncio), depois `fetch_add`/`fetch_sub` atômico.
- Release a zero → `free_registered_object` (`lib.rs:233`): desregistra,
  toma edges próprias, remove edges de entrada, chama `header.destructor`
  (`__dtor_struct/enum/tuple_*`), `libc::free`, e **depois** faz
  `ori_arc_release` de cada child das edges.
- Ciclos: `ori_arc_collect_cycles` (`lib.rs:1545`) = snapshot de **todas**
  as alocações vivas, subtrai as refs vindas de edges (trial deletion),
  marca alcançáveis a partir de `trial_count > 0`, liberta o resto.
  O(heap vivo) por passe, sob o lock global.
- Gatilhos: fim de função sync top-level, pós-`await`, explícito
  (`ori.test.collect_cycles`), e contador cooperativo de 256 allocations
  nos safe points do executor (`ORI_COOPERATIVE_COLLECT_THRESHOLD`).

## 7. Diagrama textual — destroy path (entregável C0)

### Nim (`--mm:orc`)

```text
x sai de escopo
└─ código gerado: if nimDecRefIsLastCyclicDyn(x):
   │    header.rc == 0?  não → dec; rememberCycle(x) se tipo cíclico
   │                            └─ roots.add(x); len ≥ threshold → collectCycles()
   │    sim ↓
   └─ nimDestroyAndDispose(x)
      ├─ rti = PNimTypeV2 (offset 0 do payload)
      ├─ rti.destructor(x)   ← =destroy liftado: desce campos;
      │                        cada campo ref repete o processo (cascata)
      └─ nimRawDispose(x)    ← free do bloco (header incluso)
```

Um mecanismo só: a cascata É o dtor. Ordem: campos → free do header.

### Ori (native backend)

```text
x sai de escopo
└─ codegen emitiu ori_arc_release(x)
   ├─ lock ARC_STATE → HashMap lookup (payload → header)   ← custo por op
   ├─ fetch_sub(1); antigo == 1? não → fim
   └─ free_registered_object(x, release_owned_edges=true)
      ├─ unregister_allocation(x)          (lock)
      ├─ children = take_owned_edges(x)    (lock)
      ├─ remove_incoming_edges(x)          (lock)
      ├─ header.destructor(x)   ← __dtor_*: ori_arc_release nos campos
      ├─ libc::free(header)
      └─ para cada child: ori_arc_release(child)   ← 2ª cascata (edges),
                                                      DEPOIS do free do dono
```

Dois mecanismos: dtor (campos de struct/enum/tuple) + edges
(optional/result, frames async). Ordem difere do Nim: children de edge são
liberados **após** o free do dono.

## 9. Perguntas abertas para C1 (entregável C0)

1. **Sobreposição dtor × edges** — existe algum tipo em que o codegen
   registra `__dtor_*` **e** edges para o mesmo campo? Se sim, o campo é
   retained 2× e released 2× — provar por teste (S4) ou por auditoria do
   `native_backend` que a invariante "um mecanismo por tipo" vale sempre.
2. **Enum rebind (S2 com enum)** — ao sobrescrever `var e` com outra
   variante, o `__dtor_enum_{id}` do valor antigo solta os campos da
   variante **antiga** (tag lida antes do overwrite)? O Nim resolve com
   `=destroy` por branch + `=wasMoved`; qual é a sequência exata da Ori?
3. **Ordem da cascata de edges** — a Ori libera children **depois** do
   `libc::free` do dono. Há cenário em que o dtor de um child (ou um dtor
   que aloca e registra edges durante o free) reentra no estado global de
   forma inconsistente? O Nim protege o passe do collector (`roots.len=0`
   + threshold alto); qual é a proteção equivalente da Ori em
   `free_registered_object` e dentro de `ori_arc_collect_cycles`?
4. **Verdade do layout do header** — o comentário em `lib.rs` (linha ~14)
   descreve `[ref_count: u32][type_tag: u32]`, mas o struct real é
   `AtomicI64 + Option<fn>`. O codegen depende de qual? Corrigir o
   comentário (e o spec 19-abi se citar layout) antes de mexer em memória.
5. **Custo do lock por retain/release** — o lookup em HashMap sob mutex
   global em toda op de RC é a diferença de performance mais gritante vs
   Nim. Antes de C4 (elisão): existe medição atual (benchmarks/) isolando
   esse custo? Sem baseline, elisão de RC não terá métrica de ROI.

## 10. Explicitamente NÃO fazer

- Portar `orc.nim`/`arc.nim` linha a linha para Rust.
- Adotar RC não-atômico sem antes desenhar o modelo de threads da Ori
  (eixo 6) — o Nim só pode ser não-atômico porque *move* subgrafos.
- Copiar bit-packing de cores no refcount word — só faz sentido se a Ori
  um dia trocar o full scan por buffer de suspeitos (decidir na C3).
- Jump stack / critical links (o próprio Nim desligou: ganho nulo).
- Expor `=destroy`/`=trace` ao usuário Ori.
