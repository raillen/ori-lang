# Nota Nim → Ori — C1: paridade de destrutores e cascata

- Data: 2026-07-17
- Commit Nim: `3bb46d3` (devel, clone em `_references/nim-lang/`)
- Eixo / Campanha: **C1** (Eixo 2 — destrutores/cascata; cenários S1–S4)
- Autor/agente: Claude (sessão de análise; testes exploratórios em scratchpad)

## 1. TL;DR (3 linhas)

A Ori tem **dois donos simultâneos** da liberação de campos managed (dtor
`__dtor_*` **e** edges com +1) — cada free de dono faz **release em dobro**
por campo. Provado por teste: **filho compartilhado por dois donos é
liberado com o binding ainda vivo** (use-after-free + leak, repro abaixo).
O Nim não tem essa classe: `=destroy` é o único dono; `=trace` só **lê** o
grafo, não segura referência.

## 2. O que o Nim faz (com trechos/caminhos)

- **Um mecanismo só**: o `=destroy` liftado por tipo desce os campos
  (`compiler/liftdestructors.nim`); o `=trace` gerado é usado **apenas**
  pelo collector para visitar filhos — não faz retain nem release.
- **Variantes (case objects)**: `liftdestructors.nim:174` (`nkRecCase`)
  gera `case` sobre o discriminador dentro do hook; na atribuição de
  seletor, *"the value needs to be destroyed before we assign the
  selector"* (`:178`) — destrói o payload antigo **antes** de trocar a tag.
- **Overwrite (S2)**: `x = y` vira `=copy(dest, src)`, cujo padrão canônico
  protege self-assignment (`if dest.field != source.field`) e destrói o
  destino antes de copiar (`doc/destructors.md`, seção `=copy`).
- **Ordem no free**: `nimDestroyAndDispose` chama o destructor (campos)
  e **só depois** `nimRawDispose` libera o bloco — nenhum toque no objeto
  após o free.

## 3. O que a Ori faz hoje (caminhos, verificado no código)

Base: `compiler/crates/ori-codegen/src/native_backend.rs` e
`compiler/crates/ori-runtime/src/lib.rs` (workspace 0.3.5).

- **Dtors**: `__dtor_struct/enum/tuple` fazem `ori_arc_release` de cada
  campo managed (`native_backend.rs:4775–4968`). O dtor de enum lê a tag
  **da própria alocação** (`:4870`), então rebind de variante é seguro por
  construção (a alocação antiga conserva a própria tag).
- **Literais**: struct (`:11444`), enum (`:11579`) e tuple (`:11619`)
  registram **também** `ori_arc_register_edge(base, campo)` (+1 no filho)
  — além de instalar o `__dtor_*` no `ori_alloc`. **Sobreposição total.**
- **Free do dono** (`lib.rs:233` `free_registered_object`): (1) toma as
  edges, (2) chama o dtor → release nos campos, (3) `free` do header,
  (4) release de cada child de edge → **segundo release no mesmo campo**.
- **Local rebind (S2)**: retain do novo **antes** do release do antigo
  (`:8456–8464`) — correto.
- **Field assign (S2/S4)**: `ori_arc_update_edge(owner, old, new)`
  (`:8514`) — mas o **+1 do temporário owned nunca é liberado** (não há o
  `release` que o caminho do literal tem em `:11449`).

### Contabilidade inconsistente (por que "leaks:0" engana)

| Caminho | rc do campo em regime | No free do dono (2 releases) |
|---------|----------------------|------------------------------|
| Literal (`Holder{items: xs}` com temp owned) | 1 (só edge; temp liberado) | 1º release liberta; 2º é **engolido** (campo já saiu do registry) |
| Field assign (`h.items = make_list()`) | 2 (edge + temp vazado) | os 2 releases fecham a conta **por acaso** |

Dois defeitos se cancelando. O leak check dá 0 nos dois — mas o modelo
quebra assim que um filho tem **mais de um dono** ou um binding externo.

## 4. Prova por teste (cenários exercitados)

Programas em scratchpad, `ori 0.3.5` release + `ORI_TEST_LEAK_CHECK=1`:

| ID | Cenário | Resultado | Evidência |
|----|---------|-----------|-----------|
| S1 | bind local sai de escopo | ok (suite existente `memory_arc.rs`) | leaks:0 |
| S2a | rebind de lista (`x = make_list()`), alias | ok | leaks:0 |
| S2b | enum rebind Polygon→Label→Empty | ok | leaks:0 (tag lida da alocação antiga) |
| S4b | `h.items = make_list(5)` (field assign owned) | **contabilidade errada mascarada** (rc=2 compensado pelo release duplo) | leaks:0, ver §3 |
| S4d | **filho compartilhado**: `const xs` vivo; `consume(xs)` cria e solta um `Holder{items: xs}` | **FALHA: free prematuro + UAF + leak** | `before:1 after:0` (xs liberada com binding vivo); `lists.len(xs)` pós-free = UAF; abort com 1 live allocation (exit 134) |

Repro S4d (essência):

```ori
consume(shared: list[int]) -> int
    const h: Holder = Holder { items: shared }
    return lists.len(h.items)
end
-- no chamador: xs ainda em escopo → live_allocations cai de 1 para 0
```

Causa: morte de `h` → dtor faz release(xs) **e** a edge child faz
release(xs) de novo → consome o +1 do binding do chamador.

## 5. Tabela delta

| Aspecto | Nim | Ori | Gap | Risco | Esforço |
|---------|-----|-----|-----|-------|---------|
| Dono da cascata | só `=destroy` | dtor **e** edges (+1) | release em dobro por campo | **UAF demonstrado** (S4d), janela ABA multi-thread | M |
| Grafo p/ ciclos | `=trace` (leitura, sem RC) | edges com ownership (+1) | modelos misturados | leitura do código impossível de auditar | — |
| Enum rebind | destroy antes de trocar seletor | tag na alocação antiga | nenhum | ok | — |
| Overwrite local | `=copy` c/ guarda self-assign | retain new → release old | nenhum | ok | — |
| Field assign temp owned | `=sink`/`=copy` consome | +1 do temp nunca liberado | leak mascarado pelo release duplo | vira leak real após o fix do overlap | S |
| Release engolido | não existe (sem registry) | release de ptr desregistrado é no-op silencioso | esconde double-release | mascara bugs; vira corrupção se endereço for reutilizado | — |

## 6. Ações propostas (máx 3)

### Ação 1 — Testes de regressão que capturam a classe (antes do fix)
- Por quê: S4d é reproduzível e hoje **não existe** na suite; leak check
  sozinho não pega free prematuro.
- Arquivos Ori: `compiler/crates/ori-driver/tests/memory_arc.rs` — novos
  testes: S4d (filho compartilhado, `live_allocations` before/after) e
  S4b (field assign owned, contabilidade via `live_allocations`).
- Teste: os dois falham no master atual; landar junto com a Ação 2.
- Esforço: S. Fora de escopo: mudar runtime/codegen.

### Ação 2 — ADR curto: um único dono de cascata (recomendação: edges)
- Por quê: é preciso escolher entre (A) estilo Nim — dtor é o dono, edges
  viram trace puro sem +1; ou (B) edges são o dono — `__dtor_*` param de
  fazer release de campos. **Recomendo (B)**: cirurgia menor (3 pontos:
  corpo dos `__dtor_*`, release do temp owned no field assign, conferir
  `collect_cycles`), enquanto (A) exigiria dar dtor a optional/result/
  closures/frames async que hoje só têm edges. Registrar o trade-off vs
  Nim no ADR (Nim escolheu (A); a Ori já paga o +1 das edges de qualquer
  forma).
- Arquivos Ori: `docs/planning/` (ADR) + `docs/spec/10-memory.md`.
- Teste: n/a (decisão); gate para a Ação 3.
- Esforço: S.

### Ação 3 — Implementar o dono único conforme o ADR
- Por quê: elimina a classe UAF (S4d) e o leak mascarado (S4b) de uma vez.
- Arquivos Ori (se (B)): `native_backend.rs` — `define_struct_dtors`/
  `define_enum_dtors`/`define_tuple_dtors` deixam de emitir release de
  campos (dtor pode nem ser registrado); field assign libera temp owned;
  `lib.rs` — revisar `free_registered_object`/`collect_cycles` (children
  release passa a ser o único caminho).
- Teste: Ação 1 verde + suite `memory_arc` + `ORI_TEST_LEAK_CHECK=1` +
  `performance_guard`.
- Esforço: M.

## 7. Explicitamente NÃO fazer

- Portar `=copy`/`=sink`/move semantics agora (é F4/C4 — elisão).
- "Consertar" silenciosamente mantendo os dois mecanismos com contagem +2
  por campo — dobra o custo de RC e continua ilegível.
- Remover o comportamento "release de ptr desregistrado é no-op" antes do
  fix — hoje ele é a única coisa segurando o caminho do literal.
- Mexer no layout do header (F0 cuida do comentário; Eixo 1 decide layout).
