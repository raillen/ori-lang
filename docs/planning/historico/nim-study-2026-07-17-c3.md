# Nota Nim → Ori — C3: collector incremental (suspeitos + threshold adaptativo)

- Data: 2026-07-17
- Commit Nim: `3bb46d3` (devel; referência: `lib/system/orc.nim` — roots
  buffer, `rootIdx`, threshold adaptativo; ver nota C0 §2)
- Eixo / Campanha: **C3** (Eixo 4 — cycle collector / trial deletion)
- Autor/agente: Claude (implementação nesta sessão)

## 1. TL;DR (3 linhas)

O passe cooperativo de ciclos deixou de escanear o heap inteiro: agora o
`ori_arc_release` registra **suspeitos** (decremento para >0 em objeto com
edges de saída, à la Bacon/Nim `rememberCycle`) e o passe roda trial
deletion **só no subgrafo alcançável dos suspeitos** — O(subgrafo), não
O(heap). O threshold cooperativo ficou **adaptativo por eficácia** (regra
do Nim: eficaz → encolhe ×2/3, min 64; ineficaz → cresce ×1,5, max 65 536).
`ori.test.collect_cycles`/`assert_no_leaks` mantêm o full scan.

## 2. Mapeamento Nim → Ori do que foi trazido

| Ideia Nim (orc.nim) | Implementação Ori (ori-runtime) |
|---------------------|--------------------------------|
| `rememberCycle` no decref que não zera | `ori_arc_release`: marca suspeito sob o mesmo lock quando `old != 1` e `edges.by_owner` contém o payload |
| Filtro `.acyclic.` (tipo não-cíclico não vira root) | Filtro estrutural: **só objetos com edges de saída** podem estar em ciclo — sem anotação de usuário (manifesto preservado) |
| `rootIdx` no header (remoção O(1) da lista de roots) | `suspect_idx` no `ArcAllocation` (registro, não header — sem mudar ABI) + swap-remove |
| Passe sobre `roots`, não sobre o heap | `collect_cycles_from_suspects`: fecho por edges de saída a partir dos suspeitos → `collect_cycles_over_candidates` |
| `rootsThreshold` adaptativo (`freed*2 >= touched` → reset; senão ×1,5) | `adapt_cooperative_threshold` com a mesma regra, bounds [64, 65 536]; `ORI_COOPERATIVE_COLLECT_THRESHOLD` pina (testes) |
| `GC_fullCollect` força passe completo | `ori_arc_collect_cycles` (ABI) e `ori.test.*` mantêm full scan |

## 3. O que ficou explicitamente de fora (conforme plano F3)

- **Bit-packing de cores no refcount word** (black/gray/white do Nim): o
  passe restrito por candidatos dispensa cores — `Mark { trial_count,
  marked }` temporário continua suficiente. Reavaliar só se o subgrafo de
  suspeitos ficar grande demais na prática.
- **Jump stack / critical links** (Lins): o próprio Nim desligou
  (`useJumpStack = false`, ganho nulo).
- **Atalho `rcSum == edges`** ("tudo é lixo"): fica para quando houver
  medição mostrando que vale o bookkeeping extra.

## 4. Correção (por que o passe restrito é seguro)

Edges cuja origem está **fora** do conjunto candidato não são descontadas
no trial — contam como referência externa. Logo, restringir o conjunto só
pode **sub-coletar** (deixa ciclo vivo até o próximo passe), nunca liberar
objeto alcançável. Completude: todo ciclo que virou lixo teve algum membro
decrementado para não-zero no momento da desconexão (Bacon) → esse membro
está no buffer → o ciclo inteiro está no fecho de saída dele.

## 7. Bônus da sessão — bug de ABI pré-existente achado e corrigido

O teste `compile_runs_structural_equality_advanced_native` quebrou com o
C3 ("fals" em vez de "false"). A investigação (repro mínimo →
`ORI_DUMP_CLIF` → objdump → strace → probe C) mostrou que **não era o
collector**: o Cranelift passava argumentos I8 para símbolos externos
**sem zero-extension** (`sete %dil` deixa lixo nos bits altos), enquanto o
rustc/LLVM assume a convenção SysV de caller-extension — o
`ori_bool_to_string_parts` otimizado calculava o *length* testando EDI
(lixo ⇒ len de "true") e o conteúdo testando BPL (0 ⇒ "false") →
`"fals"`. Pré-existente no master (repro mínimo
`io.println(string("a" == "b"))` falhava lá também); o C3 só mudou o
layout do heap do programa grande e expôs no teste.

**Fix:** params inteiros < 32 bits nas declarações de runtime ganham
`ArgumentExtension::Uext` (retornos ficam sem extensão — o chamador só
confia nos bits baixos, default do Cranelift). Regressão:
`compile_runs_bool_from_string_eq_prints_correctly` (ori_spec 161/161).
